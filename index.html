CHAPTER 1 — THEORY

IGA Fundamentals

Before touching OneIM, you must understand the field it operates in. Identity Governance and Administration (IGA) is a discipline — OneIM is its implementation.

🟢 Beginner — Essential Foundation

The Core Problem IGA Solves

In every organization, people join, move between departments, and eventually leave. Each person needs access to systems — email, file shares, business applications, HR systems, AD accounts, etc. Without a managed process, this creates serious problems:

⚠️
Access Sprawl
Employees accumulate more access than needed over time. When they move departments, old access is rarely removed — violating least-privilege principle.
🔍
No Visibility
IT can't answer "who has access to what?" — a basic question that becomes impossible without centralized tracking across dozens of systems.
📋
Compliance Risk
Regulations like SOX, GDPR, HIPAA require demonstrable control over who accessed sensitive data. Manual systems cannot produce this audit trail reliably.

IGA is the discipline that solves these problems — by providing a central platform to govern identities, manage access lifecycle, enforce policies, and produce compliance evidence.

IGA vs IAM — What's the Difference?

IAM — Identity & Access Management
  • Focus: Authentication and authorization (who are you, what can you do right now)
  • Tools: Active Directory, Azure AD, Okta, LDAP, SSO solutions
  • Question it answers: "Can this user log in? Can they access this resource?"
  • Time horizon: Real-time, transactional
  • Primary users: IT infrastructure teams
  • Examples: SSO, MFA, Active Directory
IGA — Identity Governance & Administration
  • Focus: Governance, lifecycle, compliance, visibility over access
  • Tools: OneIM, SailPoint, Saviynt, IBM ISIM
  • Question it answers: "Should this user have this access? Who approved it? Is it still needed?"
  • Time horizon: Lifecycle-oriented, process-driven
  • Primary users: IT security, audit, HR, compliance teams
  • Examples: OneIM, access reviews, provisioning workflows
💡
Key insight: OneIM is IGA, but it integrates WITH IAM systems (like Active Directory). OneIM reads from AD (synchronization) and writes back to AD (provisioning) — AD is a target system that OneIM governs, not a competitor to it.

The Identity Lifecycle — Joiner, Mover, Leaver

The most important concept in IGA is the identity lifecycle. Every identity goes through these stages:

  1. J
    Joiner — New Employee Arrives
    HR creates a record in the HRIS. OneIM reads this (via HR connector or CSV import). A Person record is created. Based on the person's department, job title, and location, business rules automatically determine which accounts to create and which roles to assign. The Job Server provisions these accounts in AD, Exchange, SAP, etc. — automatically, consistently, and auditably.
  2. M
    Mover — Employee Changes Role
    HR updates the employee's department or job title. OneIM detects the change. New role entitlements are calculated and provisioned. Old entitlements that don't apply to the new role are revoked. This happens automatically — the user doesn't need to submit a ticket, and IT doesn't need to remember to do it manually.
  3. L
    Leaver — Employee Departs
    HR marks the employee as terminated. OneIM immediately — or on the effective date — disables all accounts across all connected target systems, removes all group memberships, revokes all entitlements, and creates an audit record. No account is forgotten, no access lingers. This is a critical security control.
CHAPTER 2 — PLATFORM

What is One Identity Manager?

One Identity Manager (OneIM) is Quest Software's enterprise IGA platform. It is one of the leading IGA solutions in the market alongside SailPoint IdentityIQ and Saviynt.

🟢 Beginner — Essential Foundation

Platform Overview

OneIM is a modular, database-centric IGA platform that manages identity lifecycle, automated provisioning, access request workflows, compliance, and governance across the enterprise. It is installed on-premises (or in cloud VMs), with a central SQL Server database as its heart.

Unlike cloud-only SaaS IGA tools, OneIM runs in your infrastructure — giving you full control, deep customizability, and the ability to handle extremely complex enterprise requirements.

🔐
Identity Governance
Manage who has access to what. Enforce least-privilege. Run access certifications and attestation campaigns. Satisfy audit requirements.
⚡
Automated Provisioning
Automatically create, modify, or disable accounts in AD, Exchange, SAP, LDAP, Azure AD, cloud apps — driven by HR data and business rules.
🛒
IT Shop / Self-Service
Employees request access through a web portal. Approvals route automatically. Requests are tracked, auditable, and can be revoked on schedule.
📋
Compliance & Audit
Segregation of Duties rules, attestation campaigns, audit reports for SOX, GDPR, HIPAA, ISO 27001, and other regulatory frameworks.
🔗
200+ Connectors
Active Directory, Azure AD, Exchange, SAP, LDAP, Google Workspace, ServiceNow, SCIM, Oracle, Notes/Domino, and many more.
🏗️
Fully Customizable
Extend the schema, write VB.NET scripts, build custom workflows and processes. Almost any requirement can be implemented within OneIM.

OneIM Key Concepts — The Vocabulary

👤
Person (Identity)
CORE OBJECT · Central Identity Record

The Person object is the central identity record in OneIM. It represents a real human being — an employee, contractor, service account owner, etc. Person records are the source of truth for identity data. They are typically populated from an HR system or Active Directory import.

A Person is NOT the same as an account. A person can have multiple accounts across multiple systems — one AD account, one SAP account, one Exchange mailbox. All these accounts are linked back to the Person record. This linkage is what makes automated provisioning and de-provisioning possible.

🗂️
DBQueue — The Heart of Async Processing
CORE MECHANISM · Task Queue · Job Dispatch

When you change any object in OneIM (create a person, assign a role, approve a request), it doesn't immediately go and write to AD or send an email. Instead, it creates an entry in the DBQueue — a table in the SQL database that acts as a task queue.

The Job Server (One Identity Manager Service) continuously polls this queue, picks up tasks, and executes them asynchronously. This architecture makes the system resilient (tasks don't get lost), scalable (multiple Job Servers can share the load), and auditable (every task is logged).

🏢
Business Roles
CORE CONCEPT · RBAC Container · Provisioning Trigger

Business Roles are how OneIM implements Role-Based Access Control (RBAC). A Business Role is a container that groups together related entitlements (AD groups, SAP profiles, application permissions) that represent a business function (e.g., "Finance Analyst", "IT Administrator", "Sales Team Member").

When a Business Role is assigned to a Person, all the entitlements inside that role are automatically provisioned to the person's accounts. When the role is removed, the entitlements are revoked. This is the most powerful provisioning mechanism in OneIM.

🛒
IT Shop (Service Catalog)
SELF-SERVICE PORTAL · Request Management · Approval Workflows

The IT Shop is OneIM's self-service access request portal. Think of it as an "Amazon for IT access." Service items in the catalog represent things users can request: software installations, AD group memberships, business roles, file share access, etc. When a user requests an item, an approval workflow is triggered. The approver(s) review and approve or reject. Upon approval, the access is automatically provisioned.

🔄
Synchronization
DATA INTEGRATION · Read from Target · Write to Target

Synchronization is the process that keeps OneIM's database in sync with external target systems (AD, Azure AD, SAP, etc.). It reads data from the target system and imports it into the OneIM database — creating or updating Person records, account records, group memberships, etc. It can also write changes back (provisioning). Each target system has a "Synchronization Project" configured in the Synchronization Editor tool.

Step-by-Step: How a Provisioning Request Flows

  1. 1
    User Submits Access Request (Web Portal)
    User logs in to the Web Portal (Angular SPA in browser). They browse the IT Shop catalog and click "Request" on an AD group membership. The portal calls the API Server (or Application Server in older versions) via HTTPS REST.
  2. 2
    API/Application Server Processes the Request
    The Application Server authenticates the request (Windows auth, Forms auth, or OAuth). It applies security checks and business logic. It writes the new PersonWantsOrg record (the access request object) to the SQL Server database. An approval workflow entry is also created.
  3. 3
    DBQueue Entry Created Automatically
    The database write triggers a stored procedure that creates an entry in dbo.DBQueue. This entry specifies: which object changed, what task to run, and which Job Server queue to route it to. This is the core mechanism of all async processing in OneIM.
  4. 4
    Job Server Picks Up the Task
    The One Identity Manager Service (Job Server) continuously polls dbo.DBQueue. It picks up the task and executes the configured process — in this case, sending an approval email to the requester's manager.
  5. 5
    Approver Approves in Web Portal
    The manager receives an email with a link. They log in to the Web Portal, see the pending approval in "My Responsibilities → Approvals", and click Approve. This writes the approval decision back to the database.
  6. 6
    Provisioning to Active Directory
    The approval triggers another DBQueue entry — this time for the provisioning process. The Job Server picks this up, uses the Active Directory connector, connects to the AD domain controller via LDAP, and executes the equivalent of Add-ADGroupMember to add the user to the requested group. The access is now live in AD.
  7. 7
    Confirmation Email Sent
    A final DBQueue task fires the email notification process. The Job Server connects to the SMTP server and sends a "Your access request has been fulfilled" email to the requester. The entire end-to-end process is logged with timestamps in the database.

Critical Architecture Rules

⚠️
Web Portal NEVER connects directly to SQL Server. The Web Portal always goes through the Application Server (or API Server). This is a security boundary. Only the Job Server and desktop clients (Manager, Designer) connect directly to SQL in standard setups.
💡
The database is not just a data store. The OneIM database contains compiled VB.NET assemblies, stored procedures, process definitions, scripts, and virtually all business logic. When you "compile" changes in the Designer, you are compiling .NET code INTO the database. This is a unique and important architectural characteristic.
CHAPTER 4 — COMPONENTS

Components Deep Dive

Each component has a specific role. Knowing all of them — their purpose, configuration, and failure modes — is essential for any OneIM professional.

🗄️
One Identity Manager Database
SQL SERVER · Core Tier · Central Repository

The database is the absolute center of the system. Everything else is peripheral. It stores: all identity data, all account data, all entitlements, all policies, all configuration, the DBQueue, JobQueue, compiled .NET code, scripts, process definitions, and the schema metadata. If the database is unavailable, the entire system stops.

OneIM requires SQL Server with Full-Text Search (mandatory), Mixed Mode Authentication, and the SQL Server Agent running.

Critical Database Tables

  • Person — employee/identity records
  • ADSAccount — AD account objects
  • PersonWantsOrg — IT Shop requests
  • DBQueue — async task queue
  • JobQueue — job execution log
  • DialogObject — schema metadata
  • QBMVStoreProc — compiled stored procs
  • OrgHasADSGroup — role-to-group assignments
⚙️
One Identity Manager Service (Job Server)
WINDOWS SERVICE · Service Tier · Async Processing Engine

The Job Server is a Windows Service that runs in the background and processes all asynchronous tasks. It is the execution engine of OneIM. Without it running, nothing gets provisioned, no emails get sent, no sync jobs run.

It is configured via jobservice.cfg and registered in the OneIM database as a Server entry. Multiple Job Server instances can run simultaneously for load balancing and high availability — each processes tasks from its assigned queue name.

What It Does

  • Polls and processes DBQueue tasks
  • Executes provisioning to target systems
  • Sends email notifications (SMTP)
  • Runs synchronization jobs
  • Executes scheduled tasks & reports
  • Processes approval workflow steps
  • Runs compiled VB.NET process steps
  • Updates synchronization project data
🌐
Application Server
IIS WEB SERVICE · Middle Tier · REST/SOAP API Gateway

The Application Server is an IIS-hosted web service that acts as the bridge between web clients and the SQL database. All web-based clients (Web Portal, external API consumers) must go through it — they never connect directly to SQL. It handles session management, authentication, object caching, and enforces security.

It is configured via the Application Configurator tool and runs in its own IIS application pool with .NET 4.x Integrated Pipeline mode. It must be deployed and tested BEFORE the Web Portal can work.

🛒
Web Portal (IT Shop Portal)
ANGULAR SPA · Presentation Tier · End-User Interface

The Web Portal is the browser-based self-service interface for end users. In OneIM 9+, it is a modern Angular single-page application. Users access it with a browser — no software installation needed. It connects to the API Server (not directly to SQL) for all operations.

Portal Modules

  • IT Shop — browse & request access
  • My Responsibilities — manage your team
  • Attestation — certify access reviews
  • Compliance — view SoD violations
  • Service Desk — create support tickets
  • Reports — run and view reports
  • Password Reset — self-service passwords
  • Admin area — system admin tasks
🖥️
Manager (Desktop Fat Client)
WINDOWS DESKTOP · Administration · Full Object Management

The Manager is the primary Windows desktop administration tool. It provides a rich tree-based UI for browsing and editing all objects in the system: persons, accounts, groups, roles, policies, jobs, reports. Most configuration tasks in the labs use the Manager. It connects directly to the database (or via Application Server).

Think of it as "SQL Server Management Studio but for OneIM objects." It is the most-used tool for day-to-day administrative tasks.

🔬
Designer
CUSTOMIZATION TOOL · Schema Extension · Script Development

The Designer is the developer/customization tool. It allows you to: extend the database schema (add custom columns and tables), write and test VB.NET scripts, configure processes and events (automation triggers), define custom workflows, build report templates, manage system permissions, and create transport packages for migration between environments.

⚠️
Changes in Designer affect the entire system and require a compilation step. Always develop in a test environment first. Mistakes in scripts can break entire processes. Lab IM-DSE-01 covers this safely.
🔄
Synchronization Editor
CONNECTOR TOOL · Target System Integration · Mapping Engine

The Synchronization Editor is used to create and manage synchronization projects — the configuration objects that define how a target system (AD, Azure AD, SAP, etc.) connects to OneIM. Each project specifies: which objects to sync, attribute mappings, merge rules (what to do when conflicts occur), provisioning rules (what to write back), and scheduling.

It is also where you test connectivity to target systems, run ad-hoc synchronizations, and review sync errors in detail.

CHAPTER 5 — DATA MODEL

OneIM Data Model

Understanding the key objects and relationships in the OneIM database is essential for effective administration, scripting, and troubleshooting.

🟡 Intermediate — Important for Admins

Core Object Hierarchy

OneIM uses an object model where everything in the system is an object with properties. Objects are linked to other objects through foreign key relationships. The schema is defined in metadata tables and can be extended.

Key Relationships to Understand

💡
The core relationship chain: Person → PersonInOrg → Org (Business Role) → OrgHasADSGroup → ADSGroup → ADSAccount. This chain is what drives automated provisioning. When a Person is assigned to a Business Role that contains an AD Group, OneIM knows to add the person's AD account to that group.

DBQueue — How It Works in Detail

  1. 1
    Object Change Triggers the Queue
    When any object is saved in the OneIM database (INSERT, UPDATE, DELETE), database triggers fire and evaluate which processes are configured to run for that event. If a matching process is found, a new row is written to dbo.DBQueue.
  2. 2
    Job Server Polls and Locks
    The Job Server polls DBQueue every few seconds. When it finds an unprocessed task, it locks the row (sets a processing flag) to prevent other Job Server instances from picking up the same task. This is how multi-instance load balancing works safely.
  3. 3
    Process Steps Execute
    The Job Server executes each step in the process definition. Steps can be: VB.NET script calls, connector operations (write to AD), external executable calls, email sends, or sub-process triggers.
  4. 4
    Task Completes or Errors
    On success, the DBQueue row is removed (or moved to archive). On failure, the ErrorCount is incremented and the task is retried (up to a configured limit). After max retries, the task stays in the queue with an error state visible in the Manager's Job Queue monitor.

Monitoring the Job Queue in Manager

SQL · SSMS
-- Check for errored DBQueue entries SELECT TOP 20 ObjectKey, TaskName, ErrorMessage, ErrorCount, StartTime FROM dbo.DBQueue WHERE ErrorCount > 0 ORDER BY StartTime DESC; -- Count tasks by state in JobQueue SELECT JobState, COUNT(*) AS Count FROM dbo.JobQueue GROUP BY JobState; -- Find slow-running processes SELECT TaskName, AVG(DATEDIFF(second, StartTime, FinishTime)) AS AvgSeconds FROM dbo.JobQueue WHERE JobState = 'Success' AND FinishTime IS NOT NULL GROUP BY TaskName ORDER BY AvgSeconds DESC;

CHAPTER 7 — INSTALL PHASE 1

SQL Server Setup

A correctly configured SQL Server is the foundation of everything. A misconfigured SQL installation is the single most common cause of failed OneIM deployments.

PHASE 1 / STEP 1
Install SQL Server with Correct Options
  1. 1
    Run SQL Installer → New Installation
    Mount the SQL Server ISO. Run setup.exe. Click Installation → New SQL Server stand-alone installation. Accept license.
  2. 2
    Feature Selection — This Step Is Critical
    On the Feature Selection screen, you MUST select:
    • Database Engine Services — the core SQL engine (obvious)
    • Full-Text and Semantic Extractions for Search — ⚠️ MANDATORY for OneIM. If you miss this, the database installer will fail.
    • SQL Server Replication — recommended but optional
  3. 3
    Instance Configuration → Default Instance
    Choose Default Instance (MSSQLSERVER) for lab simplicity. For named instances, all OneIM connection strings will use SERVERNAME\INSTANCENAME format. Note your instance name.

Active Directory Synchronization

Connecting OneIM to Active Directory is the first major integration task. This is the foundation for all AD-based provisioning and user import capabilities.

What Synchronization Does

AD synchronization does two things: reads FROM AD INTO OneIM (importing users, groups, OUs), and writes FROM OneIM BACK TO AD (provisioning). You configure both directions in a single Synchronization Project.

Creating the AD Synchronization Project

  1. 1
    Open Synchronization Editor
    From Launchpad, click Synchronization Editor. Log in with viadmin credentials. Click New Synchronization Project. Select Active Directory as the connector type.
  2. 2
    Configure AD Connection
    Sync Editor — AD Connection
    Domain: yourdomain.local Domain Controller: dc01.yourdomain.local (use FQDN) Protocol: LDAP (or LDAPS for secure) Authentication: Windows Account: DOMAIN\svc_oneim Password: ServiceAcctPassword → Service account needs Read on all objects in scope → Write access needed only if provisioning back to AD
  3. 3
    Define Scope (Which Objects to Sync)
    LDAP Scope Example
    // Full domain scope (lab) DC=yourdomain,DC=local // Scoped to specific OU (production best practice) OU=Users,OU=Company,DC=yourdomain,DC=local
  4. 4
    Review Default Property Mappings
    Default AD → OneIM Mappings
    AD sAMAccountName → OneIM ADSAccount.AccountName AD objectGUID → OneIM ADSAccount.ObjectGUID (unique key) AD givenName → OneIM Person.FirstName AD sn → OneIM Person.LastName AD mail → OneIM Person.DefaultEmailAddress AD department → OneIM Person.Department AD title → OneIM Person.Title AD manager → OneIM Person.LineManager (resolved as FK) AD userAccountControl → OneIM ADSAccount.IsDisabled
  5. 5
    Save and Run First Full Synchronization
    Save the project. Click Start Synchronization → Full Synchronization. Monitor the progress in the Sync Editor log. After completion, open Manager → Active Directory → your domain — all users and groups should be visible.
    ✅
    Persons and AD accounts visible in Manager = synchronization successful!

📄 CSV Import

Cloud HR systems can push data via SCIM protocol to OneIM's API Server endpoint.

Person Account Linking — How It Works

  • Match by sAMAccountName = Person.CentralAccount — if a Person exists with a CentralAccount matching the AD username, the AD account is linked to that Person.
  • No match found? A new Person record is created automatically from the AD attributes (name, department, title, etc.).
  • Multiple matches? The sync project logs a conflict and does not link automatically — manual resolution needed.

IT Shop & Approval Workflows

The IT Shop is how end users request non-standard access in a controlled, auditable, self-service way.

IT Shop Structure

  • IT Shop (root): The top-level container
  • Shelf (Regal): A category of services (e.g., "Software", "AD Groups", "Business Applications")
  • Service Item: A specific thing that can be requested (e.g., "Install Adobe Acrobat", "Join Finance-Reports AD Group")
  • Product (AccProduct): The actual entitlement the service item grants when approved (an AD group, a Business Role, etc.)

Creating a Service Item

  1. 1
    Navigate to IT Shop Configuration
    In Manager: IT Shop → IT Shop → Your Shop → Shelves → [Target Shelf]. Click Assign Service Items. Or create a new AccProduct directly under IT Shop → Service Catalog → Service Items.
  2. 2
    Define What Is Granted (AccProduct)
    Link the service item to the Business Role, AD Group, or other entitlement that will be granted when approved. This is done in the Assignments tab of the service item.
  3. 3
    Configure the Approval Policy
    Set which approval policy applies to this service item. The approval policy defines the workflow: single approver, multi-level approval, auto-approve, etc. You configure approval policies under IT Shop → Approval Policies.
  4. 4
    Test the Request in Web Portal
    Log in to the Web Portal as a test user (not viadmin). Browse to IT Shop → Request. Find the service item. Submit a request. Then log in as the approver and approve it. Verify the provisioning happens in the Job Queue.

Approval Workflow Types

Workflow TypeDescriptionUse Case
Auto-ApproveRequests are automatically approved with no human interaction.Standard software, low-risk access
Single ApproverRequest goes to one defined approver (role owner, manager, or named person).Team applications, group memberships

Compliance & Attestation

Compliance features are what make OneIM a true IGA solution. They provide the evidence and controls required by SOX, GDPR, HIPAA, and other regulations.

🟡 Intermediate — Advanced Admin

Attestation (Access Certification)

  1. 1
    Create an Attestation Policy
    In Manager: Attestation → Attestation Policies → New. Define: what to attest (Person entitlements, AD group memberships, Business Role assignments, etc.), the schedule (quarterly, annually), who are the attestors, and what happens on no-response (auto-revoke or escalate).
  2. 2

  3. 3

    Audit Report Generation

Segregation of Duties (SoD)

  • Define SoD Rules: Manager → Compliance → Compliance Rules → New. Select the conflicting entitlement pairs.
  • Evaluation runs automatically: When access changes are made, OneIM evaluates SoD rules and creates Compliance Violations if conflicts are detected.
  • Violations appear in the portal: The Web Portal's Compliance module shows violations. Mitigating controls can be documented. Unmitigated violations can block access requests.
  • Risk scoring: Each rule can have a risk score. High-risk violations escalate to security review.

VB.NET Scripting in OneIM

OneIM's scripting engine uses VB.NET. Scripts live in the database, are compiled at runtime by the Job Server, and can interact with any object in the system.

🔴 Advanced — Developer Level

Scripting Environments

Value Calculations
Scripts that dynamically calculate a property value. Example: auto-generate an AD account name from FirstName + LastName rules.
Condition Scripts
Scripts that return true/false to determine if a process step should execute or if a rule applies.
Process Step Scripts
Scripts that execute as part of a Process — the main action logic. Example: make an API call to an external system, transform data.
Event Handler Scripts
Scripts that run before or after an object is saved. Example: validate data before insert, audit log on change.

Scripting API — Key Objects

VB.NET — OneIM Scripting API
' The current object being processed Dim person As IEntity = Session.Source() ' Reading a property value Dim firstName As String = person.GetValue("FirstName").String ' Setting a property value person.PutValue("CentralAccount", firstName.ToLower() & lastName.ToLower()) ' Load an object by query Dim conn As IConnection = Session.GetConnection() Dim qry As New SqlQuery("ADSAccount") qry.Where = "UID_Person = @uid" qry.AddParameter("@uid", person.GetValue("UID_Person").String) Dim adAccount As IEntity = conn.Get(qry) ' Log a message (visible in Job Queue logs) Session.LogMessage(LogSeverity.Info, "Processing person: " & firstName) ' Value calculation script - return a computed value Public Function ComputeAccountName() As String Dim src As IEntity = Session.Source() Dim first = src.GetValue("FirstName").String.ToLower() Dim last = src.GetValue("LastName").String.ToLower() Return first & "." & last End Function

Where to Write Scripts

  • Designer → Script Library — for reusable script functions that can be called from anywhere
  • Designer → Object Properties → [Object Type] → [Property] → Value Calculation — for property computation scripts
  • Designer → Processes → [Process] → [Step] → Script — for process step scripts
  • Designer → Object Events → [Event] → Script — for event handler scripts
⚠️
After writing any script in Designer: You must Compile (Ctrl+Shift+B or the compile menu) to push your changes to the database. Until you compile, your scripts are NOT active in the running system.

Q1 — What is the PRIMARY purpose of the DBQueue in OneIM?

Web Portal (end-user interface)
✅ Correct! The Synchronization Editor is the dedicated tool for creating and managing Synchronization Projects. Each project defines the connection to one target system, the object mappings, attribute mappings, and provisioning rules.
Q5 — In a OneIM deployment, can the Web Portal connect DIRECTLY to the SQL Server database?
A
Yes, it uses a direct SQL connection for performance reasons
B
Yes, but only for read operations — writes go through App Server
C
No — the Web Portal always goes through the Application Server (or API Server)
D
Only in OneIM versions 8 and below
✅ Correct! The Web Portal NEVER connects directly to SQL Server. The Application Server (or API Server in v9+) is the security boundary and middleware layer. This is a fundamental architecture rule. Only the Job Server and desktop tools (Manager/Designer) can connect directly to SQL.
Q6 — What must you do in Designer AFTER writing or modifying a script or process?
A
Restart the Job Server Windows service
B
Run a full synchronization to apply changes
C
Compile the changes (Ctrl+Shift+B or use the compile menu)
D
Refresh IIS application pool
✅ Correct! After any Designer change, you MUST compile. Compilation processes the changes into the database (generates SQL, compiles .NET assemblies). Until compiled, your Designer changes do nothing in the live system.
Q7 — What is "Attestation" in the context of IGA / OneIM?
A
The process of creating new user accounts in Active Directory
B
A method of single sign-on authentication
C
A periodic review where resource owners certify that access assignments are still correct and necessary
D
The approval workflow for new IT Shop requests
✅ Correct! Attestation (also called Access Certification) is a compliance process where managers, role owners, or resource owners review existing access assignments and certify each one as still valid — or revoke it. This is a key regulatory requirement for SOX, HIPAA, and GDPR compliance.

TROUBLESHOOTING GUIDE

Common Issues & Solutions

A systematic approach to diagnosing and fixing the most common OneIM problems encountered in labs and production environments.

Troubleshooting Framework

💡
Always start here: (1) Check the Job Queue in Manager for error tasks, (2) Check Windows Event Viewer on the OneIM server, (3) Check IIS logs if it's a web issue, (4) Check SQL Server error logs, (5) Check Sync Editor logs for sync issues.

Issue: Job Server Not Starting

Symptoms: The "One Identity Manager Service" Windows service fails to start or stops immediately.

  • Check: Windows Event Viewer → Application Log. Look for "One Identity Manager Service" errors.
  • Check: Can the service account connect to SQL? Test with SSMS using the same credentials.
  • Check: Port 1433 — is SQL Server running? Test-NetConnection -Port 1433
  • Check: jobservice.cfg — is the connection string correct? Is the queue name unique?
  • Fix: If credentials changed, update them in jobservice.cfg and in the Service Properties (Log On tab) in services.msc

Issue: Web Portal Shows Blank Page or Error

  • Check: Is the Application Server working? Browse directly to http://server/AppServer/SessionService.asmx. If that fails, fix the App Server first.
  • Check: IIS Application Pool — is it running? Not stopped due to identity error?
  • Check: IIS logs at C:\inetpub\logs\LogFiles\ — what HTTP error code appears?
  • Check: The Web Portal's appSettings.json (or web.config) — is the Application Server URL correct?
  • Fix: Restart the IIS Application Pools for both App Server and Web Portal in IIS Manager.

Issue: Synchronization Fails

  • Check: Synchronization Editor → the sync project → View Logs. The log shows the exact error and which object caused it.
  • Check: Can the OneIM server reach the AD domain controller? Test-NetConnection -Port 389 to the DC.
  • Check: Service account permissions — does it have read access to the OU being synced?
  • Check: Is the Job Server running? Sync jobs are triggered and executed by the Job Server.
  • Check: The Job Queue in Manager — is there an error entry for the sync job?

Issue: Provisioning Not Happening (AD Group Not Assigned)

  • Check: Job Queue in Manager — is there a pending or errored provisioning task?
  • Check: Is the Job Server running? Get-Service "*OneIdentity*"
  • Check: Does the server entry in Manager have the "Active Directory" component assigned under Machine Roles?
  • Check: AD connector account — does it have write permission to modify group memberships in AD?
  • Check: The Business Role chain — is the role actually assigned to the person? Is the group assigned to the role?

Issue: Database Installer Fails

  • Check: DBInstaller log file (usually in %TEMP%\OneIdentity\) — read the exact error message.
  • Most common causes: SQL Agent not running, Full-Text Search not installed, insufficient disk space on SQL Server, network interruption during the 30-60 minute install.
  • Fix: Resolve the root cause. Drop the partially created database (DROP DATABASE OneIdentityManager in SSMS). Then re-run the installer from scratch. Never try to resume a partial install.
